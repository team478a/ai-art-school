# 共通ID・紹介連携基盤 v5.7.1

## 目的

千ノ国5システム共通方針 v3 に沿って、AIアート教室から共通ID基盤へ利用者を照合し、紹介トークンを確認するための基盤を追加します。

このバージョンでは、共通ID、紹介関係、販売担当、決済、受講権、ポイントの業務定義そのものは変更しません。外部システムを正として照合結果だけを保存します。

## 動作概要

1. LINE Webhook、予約LIFF、画像生成LIFFで利用者が登録・更新されます。
2. ローカル登録は外部APIの状態にかかわらず完了します。
3. 共通ID照合と紹介確認は `integration_outbox_events` に登録されます。
4. cronが未処理イベントを再送します。
5. 外部APIから確定結果を受けた場合だけ、共通IDと紹介確認結果を保存します。

外部APIの停止やタイムアウトで、LINE登録、予約、画像生成を止めないフェイルオープン方式です。

## クライアント別設定

管理画面の「クライアント別設定」から、次の値をクライアントごとに登録します。

| 設定 | 内容 |
| --- | --- |
| 5システム連携を有効にする | 本番接続時だけ有効にします |
| 共通ID APIベースURL | 例: `https://passport.example.com` |
| プロジェクトキー | 通常は `ai-art-school` |
| HMACキーID | 接続先から発行された識別子 |
| HMAC署名シークレット | 接続先と共有する署名用秘密情報 |
| タイムアウト秒数 | 3～30秒。初期値は10秒 |

すべて `integration_` 接頭辞でテナント分離されます。別クライアントの接続情報は参照しません。

## 外部API

### 共通ID照合

- `POST /api/common-users/resolve`
- AIアート教室側会員ID、LINE user ID、テナントキー、プロジェクトキーを送信
- 応答の `common_user_id` を保存

### 紹介確認

- `POST /api/referrals/confirm`
- 紹介トークン、共通利用者ID、AIアート教室側会員IDを送信
- 外部応答が確認済みの場合のみ紹介関係を確定

リクエストはHMAC-SHA256で署名します。キーID、時刻、署名をHTTPヘッダーに付与します。

## 追加テーブル

- `integration_user_mappings`: ローカル利用者と共通IDの対応
- `integration_referral_mappings`: 紹介トークン照合結果
- `integration_outbox_events`: 外部APIへの送信待ち・再送管理

各テーブルは `tenant_id` を保持します。紹介トークンを含む送信ペイロードは処理完了後に消去します。

## 再送と障害時動作

- 最大8回まで指数バックオフで再送します。
- 連携失敗はローカルの登録処理を取り消しません。
- 失敗状態と最終エラーは連携テーブルに記録します。
- cronは有効なクライアントごとに少量ずつ処理し、1クライアントの障害を他へ波及させません。

## 導入確認

1. 連携を無効のまま更新し、従来のLINE登録・予約・画像生成を確認します。
2. テスト用クライアントだけ連携を有効にします。
3. LINE登録後、`integration_user_mappings` に待機レコードが作られることを確認します。
4. cron実行後、共通IDが解決されることを確認します。
5. `?ref=テストトークン` を付けたLIFF登録で、紹介確認が外部応答に従うことを確認します。
6. 外部APIを停止しても、ローカル登録が成功し再送待ちになることを確認します。
